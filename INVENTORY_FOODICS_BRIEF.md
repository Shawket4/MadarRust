# Brief: Foodics-grade Inventory (start here — fresh chat)

## Mission
Make the Madar **inventory** system Foodics-grade end-to-end — across the Rust backend AND the
React dashboard (and the Flutter POS where a flow requires it) — **without losing any existing
production data**. Today the inventory experience is broken on the dashboard (the UI was never
updated to a large backend overhaul that landed recently) and may have backend issues too.

**Lead with workflow and architecture, not code.** The owner cares far more about *how a cafe
manager actually uses this* than about technical detail. So the FIRST deliverables are an audit
+ **workflow wireframes**, then a phased plan. Do NOT start building before the plan is approved.

---

## ⭐ Design philosophy (this is the point)
Build for a **non-technical cafe manager**, on a laptop or tablet, who is busy. It must be
**feature-packed but simple**:
- **Plain language in the UI.** No engineering jargon. Translate internal concepts to manager
  words: "movement ledger" → "Stock history"; "weighted average cost" → just show the cost;
  "stock_count / variance" → "Count stock" / "Difference"; "epoch" → never shown.
- **Guided, task-first workflows** with strong defaults — a manager should be able to receive a
  delivery, count stock, or log waste in a few taps without reading a manual.
- **Progressive disclosure.** Show the 80% path up front; tuck advanced options (cost method,
  units-per-pack, partial counts) behind "Advanced" / expanders.
- **Fast common actions**, clear empty states, obvious "what do I do now" cues.
- **Multi-branch** aware, and **bilingual (en/ar, RTL)** like the rest of the dashboard.
- Consistent with the existing design system (Tailwind + Radix), and use the **Orval-generated
  API client** (no hand-written fetches).

---

## 📐 First deliverable: WORKFLOW WIREFRAMES (before any plan or code)
Produce **low-fidelity wireframes (ASCII/markdown boxes are fine)** plus the **click-path** for
every core cafe-manager journey. For each: the screen layout, what data is shown, the primary
action, and the empty/edge states. Keep them workflow-focused — don't sweat pixel detail.

Cover at minimum these journeys (end to end):
1. **"What do I do each morning?"** — an inventory home/dashboard: low-stock alerts, pending
   deliveries, today's waste, items needing a count, stock value at a glance.
2. **Set up an ingredient + its recipe** (link to a menu item / modifier, unit, cost, supplier,
   reorder level).
3. **Receive a delivery** — create/pick a purchase order from a supplier → receive (full or
   partial) → stock and cost update automatically.
4. **Count stock (stocktake)** — start a count (full or partial/by category) → enter counted
   quantities → see the difference vs expected → finalize (adjusts stock) → variance value.
5. **Log waste/spoilage** — pick item, quantity, reason; see today's/period waste.
6. **Transfer stock between branches.**
7. **Reorder view** — what's below par across branches, one-click "create PO".
8. **Reports a manager cares about** — current stock value, what's selling vs costing (COGS &
   margins), usage/consumption over time, waste by reason.
Also wireframe the **overall information architecture / navigation**: where Inventory lives in
the dashboard, its sub-sections, and how it connects to Menu/Recipes and Reports.

Then map each wireframe/journey to the backend it needs (existing endpoint vs. gap).

## Conventions you MUST preserve
- Money/costs in **PIASTRES** (no ×100 in backend). `cost_per_unit` is nullable: **NULL = unknown,
  never 0**.
- Ingredient base stock unit = `inventory_unit` enum (g/kg/ml/l/pcs); recipe quantities normalized
  to base unit on save.
- Multi-tenant: org → branches. Auth via `check_permission(resource, action)` over the Postgres
  `permission_resource` enum (new resource ⇒ enum rename-trick migration).
- Every endpoint needs `#[utoipa::path]`. After backend API changes: `cargo run --bin export-openapi`
  → on the dashboard `npm run generate:api` (Orval).

## What "like Foodics" means for inventory (capabilities to cover, simply)
Ingredients/items (categories, units, cost, supplier links, reorder/par levels, per-branch stock);
recipes/BOM with auto deduction on sale + COGS; purchasing (suppliers, POs, partial/full receiving,

partial) with variance reconciliation + valuation; waste with reasons + reporting; inventory
valuation, low-stock/par alerts, consumption/usage, COGS & margin reports — all multi-branch, all
presented simply.
